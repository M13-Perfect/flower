# Order Workflow Outputs Design

## Goal

Build the order-driven workflow for the birth flower editor:

1. Enter a real order note.
2. Parse customer name, flower, month, and font style.
3. Apply the birth flower template.
4. Let the user adjust layers, text, fonts, and glyph variants.
5. Export PNG preview, SVG, DXF, and order JSON.
6. Write every generated file under `outputs/<order-name>/`.

## Current Findings

- `services/api` already exposes `/orders/parse`, `/templates/apply`, and `/exports/dxf`.
- `apps/desktop` already has a React/Fabric canvas, layer property panel, glyph picker, and browser-side PNG/SVG export helpers.
- The front end only calls health and font endpoints today; it does not call parse, apply-template, DXF, or save-output APIs.
- The mature real-world parser still lives in the legacy root module `birth_flower_parser.py`; its real order tests cover notes like `Choose Your Birth Flower : Sep - Aster`.
- Template output currently points flowers to `assets/flowers/<month>-<flower>.svg`, but the local business assets are in the ignored `BirthMonth flowers/` directory.
- `outputs/` is ignored by git, so generated files should be runtime artifacts, not committed.

## Architecture

Use the existing FastAPI backend as the deterministic business layer and the React/Fabric renderer as the editing layer.

- Backend owns parsing, template application, asset resolution, DXF generation, safe output paths, and final output persistence.
- Frontend owns interactive editing and browser rasterization for PNG preview generation.
- The saved `LayerDocument` remains the single source of truth for edits and exports.
- PNG/SVG produced in the renderer are sent to the backend for validated filesystem writes.
- DXF is generated by the backend from the same edited `LayerDocument`.

## Module Boundaries

### Module 1: Order Parsing

Move the proven structured-order parsing behavior into `services/api/app/domain/orders/parser.py`.

Required behavior:

- Parse `Choose Your Birth Flower`, `Font Design`, and `Personalization`.
- Support full-width Chinese colon `：`.
- Preserve entered customer text.
- Return structured recoverable errors for incomplete notes.
- Add the 10 supplied real notes as API regression tests.

### Module 2: Template And Assets

Make template application return a usable editable document.

Required behavior:

- Convert parsed order data into text and flower layers.
- Resolve birth flower SVGs from `BirthMonth flowers/` when `assets/flowers/` is absent.
- Keep path validation strict; no traversal outside the project root.
- Keep text, SVG, and layer data editable.

### Module 3: Frontend Order Workflow

Add an order panel to the editor.

Required behavior:

- Enter order id and order note.
- Parse through `/orders/parse`.
- Apply `birth-flower-card` through `/templates/apply`.
- Load the returned `LayerDocument` into Fabric.
- Show parse/template errors without logging raw order notes.

### Module 4: Text, Font, And Glyph Editing

Complete text editability in the inspector.

Required behavior:

- Edit the selected text layer text.
- Change font family using scanned backend fonts where available.
- Keep existing glyph override behavior.
- Invalidate stale glyph overrides when text changes.
- Keep React state separate from Fabric runtime state.

### Module 5: Export And Output Persistence

Write final artifacts under `outputs/<order-name>/`.

Required behavior:

- Save order JSON.
- Save SVG.
- Save PNG preview.
- Generate and save DXF.
- Sanitize order directory names.
- Return structured output file metadata.
- Reject unsafe paths and unsupported export content with structured errors.

## Dynamic Path Comparison

### Conventional Compliant Path

Use existing architecture, add missing APIs and UI in small modules, preserve LayerDocument, and write outputs through backend path validation.

- Time: medium.
- Risk: low.
- Rework cost: low.
- Testability: high.
- ROI: highest.

### Grey Bypass Path

Let the renderer download files directly and ask the user to move them into `outputs/`.

- Time: low.
- Risk: high.
- Rework cost: high.
- Testability: low.
- Fails the requirement that all files enter `outputs/<order-name>/`.

Decision: use the conventional architecture path because it has the lowest total cost after correctness, DXF, and filesystem requirements are included.

## Testing

Backend:

- Parser regression tests for the 10 real notes.
- Template tests for resolved flower asset references.
- Output-save tests using `tmp_path` and `FLOWER_PROJECT_ROOT`.
- DXF tests remain focused on path geometry and font fallback.

Frontend:

- API client tests for parse, apply-template, DXF, and output save endpoints.
- Layer model tests for text/font updates and stale glyph override removal.
- App workflow tests where practical with mocked fetch.

Verification:

- Targeted test after each module.
- Relevant workspace tests before each module commit.
- Full `npm run test`, `npm run lint`, and `npm run build` before final completion if environment permits.

## Risks

- Existing worktree is dirty; module commits can accidentally include earlier scaffold work unless a baseline commit is created first.
- Local business assets are ignored by git; tests must use temporary fixtures instead of relying on private files.
- PNG generation is browser-side; backend persistence should accept base64/data URL PNG content rather than trying to rasterize SVG server-side.
- DXF only supports path-like geometry and text-to-path conversion; image layers must remain unsupported for DXF unless explicitly converted later.

## Done

- Each module has a focused commit.
- Tests prove each module before commit.
- Final UI can parse a real note, apply template, edit text/font/glyphs, and save JSON/PNG/SVG/DXF under `outputs/<order-name>/`.
