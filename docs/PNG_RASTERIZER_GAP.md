# PNG Rasterizer — RESOLVED (2026-06-13)

PNG rasterization is live. Batch generation now auto-detects the rasterizer:
PNG is produced whenever `cairosvg` + the native Cairo runtime are available,
and silently skipped (with reason recorded in `order.json` and CLI output)
when they are not. `--png` still forces the attempt.

## How it was resolved

- `cairosvg` 2.9.0 installed into `.venv-win` and declared in
  `requirements.txt` / `services/api/pyproject.toml` (option 1 from the
  original gap analysis).
- Native Cairo runtime installed via MSYS2:
  `pacman -S mingw-w64-ucrt-x86_64-cairo` (pulls `pixman`).
- `services/api/app/domain/exports/png.py` bootstraps DLL discovery on
  Windows: if importing `cairosvg` fails with a missing-DLL `OSError`, it
  prepends candidate directories to `PATH` and retries:
  1. `FLOWER_CAIRO_DIR` environment variable (override for non-standard setups)
  2. `C:\msys64\ucrt64\bin` (this machine's default)

## Behavior matrix

| Situation | Result |
| --- | --- |
| cairosvg + Cairo DLL available | PNG generated by default (CLI and desktop button) |
| cairosvg installed, DLL missing | auto-skip with reason; `--png` raises PNG_RASTERIZER_UNAVAILABLE |
| cairosvg not installed | same as above |

## If PNG stops working after an environment change

1. Check `python -c "from app.domain.exports.png import png_rasterizer_available; print(png_rasterizer_available())"`
2. If `False`: reinstall `cairosvg` in the venv, then verify
   `C:\msys64\ucrt64\bin\libcairo-2.dll` exists (re-run the pacman install if
   not), or point `FLOWER_CAIRO_DIR` at a directory containing the Cairo DLLs.
